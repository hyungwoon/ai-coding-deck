=== s00 | 오프닝 (284자) ===
여러분 안녕하세요. 오늘 강의의 주제는 AI 코딩 생태계입니다. AI는 어떻게 작동하고, 우리는 그것을 어떻게 쓸 수 있는지, 처음 접하는 분도 이해할 수 있도록 차근차근 풀어드리겠습니다. LLM이 텍스트를 처리하는 원리부터 시작해서, 에이전트가 뭔지, GitHub은 왜 알아야 하는지, 그리고 실제로 많은 분들이 어떤 순서로 AI 툴을 깊이 쓰게 되는지까지 다룹니다. 각 슬라이드마다 비유와 실무 맥락을 함께 드릴 테니, 끝까지 따라오시면 AI가 더 이상 블랙박스처럼 느껴지지 않을 겁니다. 시작하겠습니다.

=== s01 | LLM 정의와 토큰화 (763자) ===
LLM, 즉 Large Language Model이 뭔지부터 짚고 넘어가겠습니다. 많은 분들이 AI가 엄청난 지식을 저장하고 있는 초거대 백과사전 같은 거라고 생각하시는데요, 사실은 그렇지 않습니다. LLM의 본질은 하나입니다. 다음에 올 가장 자연스러운 단어를 확률적으로 예측하는 것. 번역도, 요약도, 코드 작성도, 추론도 — 이 하나의 원리에서 나옵니다. 지식을 저장한 게 아니라, 수천억 개의 텍스트에서 언어 패턴을 학습한 겁니다. 그렇다면 AI는 텍스트를 어떻게 처리할까요. 여러분이 입력한 문장을 AI가 그대로 읽을 것 같지만, 실제로는 먼저 토큰이라는 단위로 쪼갭니다. 토큰은 단어 전체가 아닌 의미 단위 조각입니다. 예를 들어 Claude is powerful이라는 문장은 Cl, aude, is, power, ful 이런 식으로 다섯 개 토큰으로 분해됩니다. 그리고 각 토큰은 고유한 숫자 ID로 변환됩니다. 여기서 중요한 포인트가 있습니다. 영어는 평균 네 글자가 하나의 토큰이 되는데, 한국어는 음절 단위로 더 잘게 쪼개집니다. 그래서 같은 내용을 쓰더라도 한국어는 토큰이 더 많이 소모됩니다. 비용이나 속도에 영향을 주는 실질적인 차이입니다. 토큰으로 쪼갠 뒤에는 각 숫자를 수백 차원의 벡터로 변환합니다. 이 벡터가 바로 AI 모델이 처리할 수 있는 실제 입력값이 됩니다. 결국 AI에게 텍스트를 준다는 건, AI가 이해할 수 있는 숫자 배열로 변환해서 전달한다는 뜻입니다. 이 과정이 학습, 추론, 할루시네이션 — 모든 것의 출발점입니다.

=== s02 | 벡터 임베딩과 컨텍스트 윈도우 (772자) ===
이제 벡터 임베딩이 왜 혁신적인지 얘기해 볼게요. 숫자 배열이라고 하면 그냥 ID처럼 들리는데, 사실 이 숫자들은 의미의 좌표입니다. 지도 위에 점을 찍는다고 생각해 보세요. 강아지와 고양이는 둘 다 동물이니까 벡터 공간에서 서로 가까운 곳에 위치합니다. 반면 자동차는 전혀 다른 영역에 있어요. 심지어 왕에서 남자를 빼고 여자를 더하면 여왕에 가까운 벡터가 나옵니다. 의미를 더하고 빼는 수식이 숫자로 동작하는 겁니다. 왜 이게 중요하냐면, 강아지 사진 보여줘라고 했을 때 문서에 개라고 쓰여 있어도 찾아낼 수 있습니다. 표현이 달라도 의미가 같으면 벡터 공간에서 가까우니까요. 이것이 시맨틱 서치, 즉 의미 기반 검색입니다. 키워드가 달라도 정확히 찾을 수 있는 이유가 바로 이겁니다. 다음은 컨텍스트 윈도우입니다. AI가 한 번에 펼쳐볼 수 있는 책장이라고 생각하세요. 책장 안에 든 것만 참고할 수 있고, 책장 밖에 있는 것은 AI에게 존재하지 않는 것과 같습니다. Claude의 책장은 최대 200K 토큰입니다. A4 용지로 약 500장 분량이에요. 이 안에 시스템 프롬프트, 업로드한 문서, 대화 히스토리, 현재 질문이 모두 들어갑니다. 대화가 길어질수록 오래된 내용이 책장 밖으로 밀려납니다. AI가 앞서 말씀하신 내용을 갑자기 모르는 것처럼 보이는 이유가 이겁니다. 컨텍스트 윈도우에서 밀려난 거죠. 외부 지식을 벡터 DB에 저장하고 검색해서 컨텍스트에 주입하는 패턴을 RAG라고 부릅니다. 이 방법으로 최신 정보나 내부 문서를 AI가 참고하도록 만들 수 있습니다.

=== s03 | 어텐션 추론과 스트리밍 출력 (780자) ===
이번엔 AI가 어떻게 추론하고 답변을 생성하는지 살펴보겠습니다. 핵심 개념은 어텐션입니다. 형광펜으로 문장을 읽는다고 상상해 보세요. 그가 공을 찼다라는 문장에서 의미를 파악하려면 그와 찼다를 강하게 연결해야 합니다. 누가 찼는지 알아야 하니까요. 공을은 찼다와 중간 정도로 연결되고요. AI는 모든 토큰 쌍에 대해 얼마나 관련 있는지를 수치로 계산합니다. 이 수치를 어텐션 가중치라고 합니다. 그리고 이 계산을 수백 개의 레이어에서 반복합니다. 이 과정이 문맥 속 의미를 이해하는 방법입니다. AI가 생각하는 게 아니라, 관계를 계산하는 겁니다. 추론이 끝나면 다음에 올 토큰의 확률 분포가 나옵니다. 예를 들어 오늘 날씨는 다음에 맑다가 40퍼센트, 흐리다가 30퍼센트, 더울 것이 20퍼센트 이런 식이에요. 가장 높은 확률의 토큰을 하나 선택하고, 그 토큰을 컨텍스트에 추가한 다음, 다시 추론합니다. 이 과정을 반복하면서 한 글자씩 생성합니다. ChatGPT나 Claude에서 답변이 흘러나오듯 표시되는 이유가 이 구조 때문입니다. 한꺼번에 계산하는 게 아니라 토큰 하나씩 만들어내는 거예요. 여기서 할루시네이션이 왜 발생하는지도 이해가 됩니다. AI는 사실을 저장한 게 아니라 패턴을 예측합니다. 모르는 것도 패턴에 맞게 그럴듯하게 지어낼 수 있는 겁니다. 중요한 정보는 반드시 검증이 필요한 이유입니다. 프롬프트를 어떻게 쓰느냐가 이 확률 분포 자체를 바꿉니다. 역할을 부여하거나, 예시를 주거나, 단계를 나눠 생각하게 하면 같은 모델에서 전혀 다른 품질의 결과가 나오는 이유입니다.

=== s04 | LLM vs AI 에이전트 (579자) ===
LLM과 에이전트의 차이를 확실히 짚고 넘어가겠습니다. 계산기와 로봇을 비교해 보세요. 계산기에 숫자를 입력하면 결과가 나옵니다. 기억도 없고, 다음 행동도 없습니다. LLM 단독이 이렇습니다. 질문이 들어오면 답변이 나옵니다. 매번 처음부터 시작하고, 결과를 받아 다음에 무엇을 할지는 사람이 판단하고 실행해야 합니다. 반면 에이전트는 목표를 주면 스스로 계획을 세우고, 실행하고, 결과를 보고, 잘못됐으면 재시도합니다. LLM에 툴을 연결하면 에이전트가 됩니다. 여기서 툴이란 파일을 읽고 쓰는 것, 터미널 명령어를 실행하는 것, 웹을 검색하는 것, API를 호출하는 것들입니다. 에이전트의 구성 요소는 크게 다섯 가지입니다. 첫째, 판단과 추론을 담당하는 LLM 자체. 둘째, 파일과 터미널과 웹을 실행하는 툴 호출. 셋째, 실행하고 결과를 확인하고 재시도하는 루프. 넷째, 대화와 작업 맥락을 유지하는 메모리. 다섯째, 외부 앱을 연결하는 프로토콜인 MCP입니다. 왜 이게 게임체인저냐면, 사람이 결과를 받아서 직접 실행하던 단계가 사라집니다. 목표만 주면 됩니다. 이 차이가 단순한 도구에서 자율 실행 시스템으로의 전환입니다.

=== s05 | GitHub 기초 (729자) ===
GitHub 얘기를 왜 하냐고 물으실 수 있어요. 코딩을 전혀 안 하는 분도 이제 AI 에이전트가 파일을 직접 만들고 수정하는 시대에 살고 있습니다. 에이전트가 작업한 결과물이 어떻게 관리되는지 이해해야 제대로 쓸 수 있습니다. 네 가지 개념만 알면 됩니다. 첫째, Repository입니다. 레포지토리는 프로젝트 폴더 전체입니다. 파일과 폴더와 변경 이력이 통째로 들어 있는 프로젝트 상자입니다. Claude Code는 이 레포를 통째로 읽고 작업합니다. 둘째, Commit입니다. 세이브포인트라고 생각하세요. 게임에서 저장하듯이, 지금 이 상태를 기록해 두는 겁니다. AI가 작업할 때마다 커밋을 남기면 실수해도 언제든 그 시점으로 돌아올 수 있습니다. 셋째, Branch입니다. 평행우주입니다. 원본은 건드리지 않고 복사본에서 실험하는 공간이에요. 기능A 브랜치에서 AI가 마음껏 작업하고, 잘 되면 메인에 합치는 방식입니다. 실패해도 원본은 안전합니다. 넷째, Pull Request입니다. PR이라고 줄여서 부릅니다. 브랜치 작업이 끝나면 메인에 합쳐달라는 검토 요청입니다. 변경 내용이 한눈에 보이고, 사람이 확인한 뒤 승인하면 합쳐집니다. 실제 AI 에이전트와 함께 일하는 흐름은 이렇습니다. 새 브랜치를 만들고, Claude Code에게 태스크를 시키고, AI가 작업하면서 커밋을 남기고, 내가 결과를 검토하고, PR을 열어서 머지하고, 배포합니다. 사람은 결정 단계에만 개입하면 됩니다.

=== s05b | 개발의 기본 구조 (781자) ===
식당을 상상해 보세요. 손님이 앉아 있는 홀, 음식을 만드는 주방, 그리고 재료가 쌓여 있는 냉장고. 모든 앱은 정확히 이 세 가지로 이루어져 있습니다.

첫 번째는 프론트엔드입니다. 사용자가 직접 눈으로 보고 손가락으로 누르는 화면이죠. 브라우저에서 실행되는 HTML, CSS, JavaScript가 여기에 속하고요. React나 Next.js 같은 프레임워크는 이 화면을 더 효율적으로 만들어 주는 도구입니다.

두 번째는 백엔드입니다. 눈에는 보이지 않지만 실제 일이 일어나는 곳이에요. 로그인을 처리하고, 결제를 검증하고, 추천 알고리즘을 돌리는 서버가 여기에 있습니다. Node.js, Python, Go 같은 언어가 많이 쓰입니다.

세 번째는 데이터베이스입니다. 앱의 기억 장치예요. 회원 정보, 주문 내역, 게시글 같은 데이터를 저장하고 꺼내오는 역할을 합니다. PostgreSQL 같은 SQL 데이터베이스가 가장 많이 쓰이고, Supabase처럼 백엔드와 DB를 묶어서 제공하는 서비스도 있습니다.

이 세 층이 어떻게 연결되는지 보면, 사용자가 버튼을 누르면 프론트엔드가 백엔드에 API 요청을 보내고, 백엔드는 데이터베이스에서 필요한 정보를 꺼내서 프론트엔드에 돌려줍니다. 그러면 화면이 업데이트되죠.

이게 왜 중요하냐고요? AI 코딩을 할 때, Claude Code는 이 전체 구조를 통째로 이해하고 작업합니다. 프론트엔드만 고치는 게 아니라, 필요하면 백엔드 API를 추가하고 DB 테이블도 함께 만들어 줍니다. 구조를 알면 AI에게 더 정확한 지시를 내릴 수 있습니다.

=== s05c | API, CLI, MCP (885자) ===
AI 코딩을 하다 보면 세 가지 용어가 반복해서 등장합니다. API, CLI, 그리고 MCP입니다. 각각 뭔지 비유로 먼저 이해해 봅시다.

API는 식당의 메뉴판이라고 생각하세요. 손님이 메뉴판에 나온 대로 주문하면, 주방에서 그대로 만들어 줍니다. 프로그램 사이에도 이런 규칙이 필요하거든요. "이 주소로 이 형식으로 요청을 보내면 이 결과를 돌려줄게." 이게 API입니다. OpenAI API를 호출한다는 건, OpenAI가 정해놓은 메뉴판 형식으로 AI 모델에게 요청을 보내는 거예요.

CLI는 직접 명령하는 리모컨입니다. 마우스로 클릭하는 대신, 텍스트로 컴퓨터에게 직접 명령을 내립니다. 터미널에서 "claude"를 입력하면 Claude Code가 실행되고, "git commit"을 입력하면 코드가 저장됩니다. AI 에이전트는 이 CLI를 통해 실제로 명령을 실행합니다. 코드를 작성하고, 파일을 열고, 서버를 실행하는 모든 게 CLI를 거칩니다.

마지막으로 MCP는 AI의 만능 어댑터입니다. 이게 진짜 게임 체인저입니다. 기존에는 AI가 외부 앱과 연동하려면 각각의 API를 직접 연결해야 했어요. Notion API 따로, Slack API 따로, GitHub API 따로. 그런데 MCP는 이 모든 걸 하나의 표준 규격으로 통일합니다. Claude Code에 MCP를 연결하면 Notion 문서를 직접 읽고 쓰고, Slack 메시지를 보내고, GitHub 코드를 관리하는 게 다 됩니다.

왜 중요한가요? 지금 여러분이 쓰는 Claude Code는 CLI 위에서 돌아가고, API로 AI 모델을 호출하며, MCP로 외부 도구와 연결됩니다. 이 세 개념을 이해하면, AI가 어떻게 실제로 작업하는지 그림이 보입니다. 그리고 더 강력하게 활용할 수 있게 됩니다.

=== s06 | AI 코딩 생태계 전체 관계도 (765자) ===
이제 전체 그림을 보겠습니다. AI 코딩 생태계는 다섯 개의 레이어로 이루어져 있습니다. 건물로 비유하면 기초부터 꼭대기까지입니다. 맨 아래 기초는 터미널 환경입니다. Tmux, iTerm2, Warp, zsh 같은 것들이에요. 에이전트가 실제로 명령어를 실행하는 바닥입니다. 이 위에 에디터와 IDE가 올라옵니다. Cursor, Windsurf, VS Code, Zed, JetBrains 같은 것들입니다. 사람이 코드를 보고 수정하는 주 공간이에요. 여기까지는 사람이 주도하고 AI가 보조하는 영역입니다. 그 위에 에디터 내 에이전트 레이어가 있습니다. Cursor의 Agent Mode, Windsurf의 Cascade, Cline, GitHub Copilot 같은 것들입니다. 에디터 안에 에이전트 기능이 탑재된 형태입니다. 사람과 AI가 함께 주도하는 하이브리드 방식입니다. 더 위에 코딩 에이전트가 있습니다. Claude Code, Devin, Aider 같은 것들입니다. AI가 주도하고 사람은 태스크만 줍니다. 자율 실행 영역입니다. 그리고 맨 꼭대기에 AI 모델이 있습니다. Claude, GPT-4o, Gemini, 오픈소스 모델들입니다. 모든 레이어의 도구들이 결국 이 모델을 호출해서 동작합니다. 위에서 아래로 봤을 때, 모델이 에이전트를 구동하고, 에이전트가 에디터와 터미널 위에서 실행되는 구조입니다. 이 전체 구조를 이해해야 도구를 선택할 때, 왜 이 도구를 쓰는지 맥락이 잡힙니다. 내가 지금 어느 레이어에서 일하는지 파악할 수 있거든요.

=== s07 | AI 툴 진화 경로 — 초기 단계 (749자) ===
마지막으로, 실제로 많은 분들이 AI 툴을 어떤 순서로 깊이 쓰게 되는지 얘기해 보겠습니다. 대부분이 이 세 단계를 거칩니다. 첫 번째 단계는 단독 LLM 문답입니다. ChatGPT나 Claude.ai, Gemini 브라우저에 들어가서 질문하고, 답을 복사해서 붙여넣는 단계입니다. 브라우저만 있으면 됩니다. 이 단계의 아하 모먼트는 와, 이게 진짜로 되네 하는 순간이에요. 그런데 얼마 지나면 불편함이 생깁니다. 매번 같은 작업을 반복하는 게 귀찮아집니다. 그래서 자동화 욕구가 생기죠. 두 번째 단계는 노코드 자동화 연결입니다. Zapier나 Make 같은 도구로 Gmail에서 AI 요약이 오면 Slack으로 자동 전송 같은 플로우를 만듭니다. 코딩 없이 앱들을 연결합니다. 이 단계의 아하 모먼트는 내가 아무것도 안 해도 알아서 돌아가네 하는 감각입니다. 그런데 정해진 흐름 말고, 상황에 따라 AI가 판단도 해줬으면 하는 욕구가 생깁니다. 세 번째 단계가 Claude Code와 터미널 입문입니다. 말로 태스크를 시키면 AI가 파일과 폴더를 직접 만들고 수정합니다. 처음엔 터미널이 낯설고 무섭지만, 결과가 나오니까 계속하게 됩니다. 이 단계의 아하 모먼트는 내가 코드를 모르는데 AI가 코드를 짜고 실행까지 했다 하는 순간입니다. 이 세 단계를 지나고 나서 더 깊은 워크플로우로 진화하게 됩니다. 이 강의에서 다룬 LLM의 원리, 에이전트 구조, GitHub, 생태계 전체 그림이 바로 그 다음 단계를 이해하기 위한 기반입니다.

=== s08 | 진화 심화 — Stage 4-6 (863자) ===
앞 슬라이드에서 Stage 1부터 3까지, 기초 환경을 구축하는 흐름을 보셨죠. 이제 그 다음 단계입니다.

Stage 4는 AI 네이티브 터미널 환경입니다. 여러 Claude Code 세션을 탭과 패널로 동시에 운용하는 단계예요. 세션마다 별도 컨텍스트를 유지하면서, 메모리를 효율적으로 분리합니다. 여기서 등장하는 도구가 Warp입니다. Warp는 AI가 네이티브로 통합된 터미널인데요, 기존에 많이 쓰던 iTerm2와 Tmux 조합 없이도 동일한 멀티세션 환경을 구현할 수 있습니다. 자연어로 명령어를 입력할 수도 있고요. 이 단계가 완성되면 단일 에이전트의 한계가 보이기 시작합니다.

그래서 Stage 5에서는 멀티 에이전트와 하네스로 넘어갑니다. 하나의 에이전트가 혼자 모든 걸 처리하는 게 아니라, 여러 에이전트를 목적별로 조합해서 씁니다. 예를 들면 리서치는 A 에이전트, 코드 작성은 B 에이전트, 테스트는 C 에이전트 이런 식으로 역할을 나누는 거죠. 그리고 이 에이전트들을 묶어서 오케스트레이션하는 구조가 바로 하네스입니다. 어떤 태스크를 어떤 에이전트에게 넘길지, 흐름 전체를 설계하는 단계예요.

마지막 Stage 6는 MCP, Model Context Protocol입니다. MCP는 Claude Code가 Notion, Slack, Gmail, GitHub, Figma 같은 외부 앱을 직접 읽고 쓸 수 있게 해주는 연결 프로토콜이에요. Zapier 같은 노코드 자동화 도구 없이도, AI 에이전트가 앱 간의 판단과 실행을 직접 처리합니다. 슬라이드에 이런 표현이 있는데요, Stage 2의 노코드 자동화가 에이전트 레벨로 업그레이드되는 지점이라고 합니다. 이 세 단계를 거치면 비로소 에이전트 생태계가 완성됩니다.

=== s09 | Claude Code 핵심 개념 (916자) ===
Claude Code를 처음 접하시는 분들이 가장 많이 하시는 오해가 있어요. "아, Claude 쓰는 터미널 앱이구나" 하고 넘어가시는 거죠. 근데 사실 Claude Code는 단순한 CLI 도구가 아닙니다. 작업 흐름 전체를 설계할 수 있는 프레임워크예요. 이 차이가 어디서 오느냐, 6가지 핵심 개념에서 옵니다.

첫 번째, CLAUDE.md입니다. 프로젝트 루트에 두는 에이전트 행동 지침서예요. "TypeScript만 써", "테스트 먼저 작성해", "main 직접 푸시 금지" 이런 규칙을 여기 적어두면, 매번 설명하지 않아도 됩니다. 새 직원에게 업무 매뉴얼을 건네는 것과 같습니다.

두 번째, 서브 에이전트입니다. 메인 에이전트가 하위 태스크를 별도 에이전트에 위임하는 구조예요. 병렬 실행으로 속도도 빨라집니다. 팀장이 팀원들에게 업무를 나눠주는 것과 같습니다.

세 번째, 커맨드입니다. 슬래시로 시작하는 단축 명령어예요. /review 하면 코드 리뷰, /deploy 하면 배포 체크리스트, 자주 쓰는 프롬프트를 한 단어로 호출합니다.

네 번째, 훅, Hooks입니다. 특정 이벤트 발생 시 자동으로 실행되는 스크립트예요. 파일 저장 후 린트, 커밋 전 테스트, 작업 완료 후 Slack 알림, 이런 식으로 자동화됩니다.

다섯 번째, 스킬, Skills입니다. 반복 작업 방식을 파일로 정의해두는 재사용 모듈이에요. 코드 리뷰 방법, 문서 작성 방법을 스킬 파일에 저장해두고 필요할 때 불러씁니다.

마지막 여섯 번째, MCP 서버입니다. Model Context Protocol의 줄임말로, Claude Code가 Notion, Slack, GitHub 같은 외부 앱을 직접 읽고 쓸 수 있게 해주는 연결 프로토콜입니다. 이 여섯 가지가 맞물릴 때 비로소 Claude Code가 진짜 프레임워크로서 작동합니다.

=== s10 | 바이브 코딩 루프 (943자) ===
"플래닝이 90%다." 이 말을 처음 들으시면 좀 과장된 것 같죠? 실제로 써보면 이게 사실이라는 걸 알게 됩니다. 왜 그런지 설명해 드릴게요.

바이브 코딩 루프는 5개 페이즈로 돌아갑니다. Phase 1은 준비 단계입니다. CLAUDE.md를 작성하고, 공식 문서를 등록하고, 스킬 파일을 세팅하는 거예요. 이 기반이 없으면 이후 플래닝도 허공에 뜨게 됩니다.

그리고 가장 중요한 Phase 2, 플래닝입니다. 슬라이드에 별표가 표시된 이유가 여기 있어요. 요구사항을 구체화하고, 단계를 분해하고, 엣지케이스를 정의하는 단계입니다. 여기서 시간을 많이 쓸수록 좋습니다. 역설적이죠? 빨리 만들고 싶은데 오히려 여기서 멈추라니까요. 근데 이유가 있어요. Phase 2에서 생각을 충분히 했다면, Phase 4 구현 단계에서 AI가 거의 혼자 다 합니다. 플래닝에 30분 쓰면 구현에 10분, 플래닝을 5분 만에 끝내면 구현 중에 계속 방향이 꺾입니다.

Phase 3은 플랜 점검입니다. AI에게 플랜을 출력하라고 요청하고, 방향이 맞는지 사람이 확인하는 단계예요. 구현 전에 반드시 멈춰야 하는 체크포인트입니다. 여기서 방향이 틀렸다면 이미 구현 들어가기 전에 고칠 수 있으니까요.

Phase 4가 구현입니다. 이 단계에서 여러분이 해야 할 일은 지켜보는 것입니다. AI가 코드를 작성하고, 필요하면 서브 에이전트에게 위임도 합니다. 개입은 최소화하는 게 좋아요.

Phase 5는 점검과 개선 단계입니다. 의도대로 됐는지 확인하고, CLAUDE.md와 README를 업데이트합니다. 그리고 다시 Phase 2로 돌아가는 루프입니다.

팁 하나 더 드릴게요. 플래닝 단계에서 프롬프트에 think 또는 ultrathink라는 단어를 넣으면 효과가 있습니다. Claude가 실행하기 전에 더 깊게 추론하도록 유도하는 키워드예요. 복잡한 태스크일수록 이 차이가 명확하게 나타납니다.

=== s11 | 바이브 코딩 안티패턴 (886자) ===
이제 실수 이야기를 해볼게요. 경험자들이 반복해서 겪는 패턴 7가지입니다. 미리 알면 절반은 피할 수 있습니다.

첫 번째이자 가장 비싼 실수입니다. 플래닝 없이 바로 구현 요청. "로그인 기능 만들어줘"처럼 모호하게 던지면 어떻게 되냐면, AI가 엉뚱한 방향으로 달려갑니다. 그리고 나중에 다 뜯어고쳐야 해요. 무엇을, 왜, 어떻게를 먼저 정의하고 플랜을 확인한 다음에 구현을 지시하세요.

두 번째는 컨텍스트 오염입니다. 같은 세션에서 너무 많은 수정을 반복하면, AI가 앞뒤 맥락에 혼란을 느끼기 시작합니다. 답변 품질이 눈에 띄게 떨어져요. 큰 방향이 바뀌는 순간 새 세션을 시작하고, CLAUDE.md로 맥락을 다시 주입하세요.

세 번째는 무한 수정 루프입니다. "조금만 더 고쳐줘"를 반복하다 보면 코드가 점점 복잡해지고, 나중엔 리팩터링도 불가능해집니다. 일정 수준 이상의 변경은 처음부터 다시 짜는 게 오히려 더 빠릅니다.

네 번째, 검증 없이 배포. AI가 "완성됐다"고 해도 직접 실행해보셔야 합니다. AI 출력은 초안입니다. 완성본이 아니에요. 이 마인드셋이 정말 중요합니다.

다섯 번째, 이해 못 한 채로 사용. AI가 만든 코드를 블랙박스로 두면 버그가 났을 때 고칠 수가 없습니다. "이 코드 한 줄씩 설명해줘"라고 항상 물어보며 진행하세요.

여섯 번째, 문서 업데이트 생략. 구현 후 CLAUDE.md를 업데이트하지 않으면, 다음 세션의 Claude가 이전 상황을 모른 채 시작합니다. 구현과 문서 업데이트는 세트입니다.

일곱 번째이자 마지막, 너무 큰 태스크를 한 번에. "전체 앱 만들어줘"처럼 큰 요청은 중간에 방향이 틀어지면 수정 범위가 너무 넓어집니다. 기능 단위로 잘게 쪼개서 각 단계를 확인하며 진행하세요. 커밋도 작게, 자주 하세요.

=== s12 | 하네스 — AI 에이전트를 길들이는 구조 (881자) ===
하네스라는 단어, 처음 들어보시는 분들도 계실 텐데요. 원래 말에 채우는 고삐와 마구를 뜻합니다. 말의 힘 자체는 건드리지 않습니다. 방향만 잡아주는 거예요. AI 에이전트에 이 비유를 그대로 적용할 수 있습니다.

에이전트는 강력합니다. 하지만 방향 없이는 쓸모없습니다. Claude가 아무리 뛰어난 모델이라도, 내 프로젝트가 뭔지, 어떤 코딩 스타일을 써야 하는지, 어디에 접근 가능한지 모르면 제대로 일을 못해요. 하네스 없는 에이전트의 모습을 보면, 매번 "우리 프로젝트는 이런 구조예요"부터 설명해야 합니다. 코딩 스타일이 매번 달라지고, 해서는 안 되는 실수를 반복하고, 외부 도구에는 접근도 못하고, 세션이 바뀌면 모든 맥락이 리셋됩니다. 고삐 없는 말이 아무리 빠르게 달려도 엉뚱한 방향으로 가는 것과 같습니다.

반면 하네스가 있는 에이전트는 어떻냐면, CLAUDE.md가 프로젝트 맥락을 자동으로 전달합니다. 스킬 파일이 일관된 코딩 패턴을 유지시켜 줍니다. 규칙이 금지 사항과 제약 조건을 강제합니다. MCP로 데이터베이스, API, 외부 도구를 직접 조작할 수 있고, 온톨로지가 장기 기억 역할을 해서 맥락이 유지됩니다. 조련된 경주마처럼, 강력한 힘이 정확한 방향으로 발휘되는 거죠.

정리하면 이렇습니다. CLAUDE.md는 지시서입니다. 에이전트에게 역할과 규칙을 알려주는 문서예요. 스킬과 규칙은 습관입니다. 반복 작업 패턴을 파일로 저장해서 일관성을 자동으로 유지합니다. 그리고 MCP와 온톨로지는 감각과 기억입니다. MCP는 에이전트의 손이고, 온톨로지는 장기 기억이에요. 이 둘이 합쳐져야 에이전트가 진짜 일을 한다고 볼 수 있습니다. 앞서 다룬 모든 개념, CLAUDE.md, 스킬, MCP, 온톨로지가 바로 이 하네스의 구성 부품입니다.

=== s12b | 코딩 하네스 실전 (825자) ===
건축 현장을 떠올려 보세요. 현장 감독 한 명이 전체 공사를 지휘하고 있습니다. 전기 팀, 배관 팀, 인테리어 팀에게 각각 작업을 배분하죠. 그리고 각 팀원은 레이저 가이드가 달린 정밀 공구로 정확하게 시공합니다.

AI 코딩 하네스가 정확히 이 구조입니다.

"로그인 기능 만들어줘." 이 한 마디를 하네스 없는 AI에게 던지면 어떻게 될까요? AI가 혼자 판단해서 대충 만듭니다. 설계 없이, 테스트 없이.

하네스가 있으면 달라집니다. 현장 감독 역할은 OMC, oh-my-claudecode가 맡습니다. OMC는 이 요청을 받는 순간 분해합니다. DB 설계, API 구현, 화면 작업, 테스트 작성. 네 가지 작업으로 쪼개서 각각 전문 에이전트에게 위임합니다. 단순한 작업은 저렴한 Haiku 모델로, 복잡한 아키텍처 결정은 비싼 Opus 모델로 보냅니다. 비용과 품질을 동시에 잡는 거예요.

정밀 공구 역할은 hwclaude가 맡습니다. 파일을 수정할 때 줄 번호와 해시값으로 정확한 위치를 찍어서 편집합니다. 이 방식 도입 전에는 파일 편집 성공률이 6.7%에 불과했습니다. 도입 후엔 68.3%까지 올라갔어요. 열 배 이상의 차이입니다. 편집이 실패하면? 원인을 분석해서 복구 가이드를 자동으로 AI에게 주입합니다. 들여쓰기가 깨지거나 줄이 합쳐지는 흔한 실수도 자동으로 잡아냅니다. 저장할 때마다 문법과 타입 오류도 검사하고요.

역할이 명확합니다. OMC는 무엇을 어떤 순서로 할지를 결정합니다. hwclaude는 그걸 얼마나 정확하게 실행할지를 담당합니다. 둘이 함께 작동하면, "로그인 만들어줘" 한 마디가 DB부터 UI까지 완성된 기능으로 돌아옵니다.

이쯤 되면 이런 생각이 드실 겁니다. 하네스는 코딩할 때만 쓰는 거 아닌가요? 아닙니다. 하네스는 구조입니다. 어떤 도메인에도 씌울 수 있습니다.

=== s13 | 하네스 성숙도 (778자) ===
잠깐 여러분 자신을 돌아보는 시간을 가져볼게요. 지금 여러분은 어디에 계신가요?

Level 0은 무장비 단계입니다. ChatGPT에 질문만 하는 상태예요. 매번 "나는 개발자인데, 프로젝트는 이렇고, 이번에 하려는 건 이거야"부터 설명합니다. 하네스가 전혀 없는 상태입니다.

Level 1은 기본 제어 단계입니다. CLAUDE.md를 작성해서 프로젝트 맥락을 전달하기 시작한 상태예요. 맥락 하네스라고 부릅니다. 이것만 해도 체감 차이가 큽니다. 매 세션마다 설명을 반복하지 않아도 되니까요. 오늘 강의 끝나고 바로 적용해볼 수 있는 첫 번째 단계입니다.

Level 2는 도구 연결 단계입니다. MCP로 외부 도구를 연동하고, 스킬 파일과 규칙을 추가한 상태예요. 행동 하네스라고 합니다. 에이전트가 실제로 외부 앱과 상호작용하고, 일관된 패턴으로 작업합니다.

Level 3은 자율 순환 단계입니다. 온톨로지와 RAG, Retrieval Augmented Generation을 도입해서 피드백 루프를 운영하는 단계예요. 풀 하네스라고 부릅니다. 에이전트가 장기 기억을 가지고, 스스로 지식을 축적하며 작업합니다.

그런데 중요한 포인트가 있습니다. 모두가 같은 Claude를, 같은 GPT를 씁니다. 모델은 공유됩니다. 하지만 어떤 하네스를 씌웠느냐에 따라 결과가 완전히 달라집니다. 잘 만든 하네스는 복제 불가능한 경쟁 우위입니다. 같은 도구를 쓰더라도 훨씬 더 좋은 결과를 만들 수 있다는 거예요. Level 0에서 Level 1로 올라가는 것만으로도 충분히 시작할 수 있습니다.

=== s14 | 에디터 vs 에이전트 핵심 차이 (854자) ===
이제 도구 선택 기준에 대해 이야기해 보겠습니다. VS Code나 Cursor 같은 에디터를 쓰는 것과, Claude Code 같은 순수 에이전트를 쓰는 것, 어떻게 다른지 명확하게 구분해 드릴게요.

핵심은 주도권이 누구에게 있느냐입니다. 스펙트럼으로 보시면 됩니다.

첫 번째 계층, 에디터입니다. VS Code, Zed 같은 도구예요. 작업 단위가 줄 단위, 함수 단위입니다. AI가 제안을 하면 사람이 매 스텝마다 확인하고 적용합니다. 터미널 실행 권한은 없고, 주도권은 완전히 사람에게 있습니다. AI가 보조자 역할이에요.

두 번째 계층, 에디터 내 에이전트입니다. Cursor Agent Mode가 대표적인 예인데요, 중간 지대라고 보시면 됩니다. 작업 단위가 피처, 태스크 수준으로 올라갑니다. 사람은 승인만 합니다. 터미널 실행은 제한적으로 가능하고, 주도권이 사람에서 AI 쪽으로 이동하기 시작합니다.

세 번째 계층, 순수 에이전트입니다. Claude Code가 여기에 해당합니다. 작업 단위가 태스크, 레포 전체 수준입니다. 사람은 시작과 끝만 관여합니다. 터미널 실행 권한이 풀로 주어지고, 주도권이 AI에게 있습니다.

그러면 언제 어떤 걸 써야 할까요? 줄 단위로 세밀하게 제어하고 싶을 때, 코드를 직접 보면서 작업하고 싶을 때는 에디터입니다. 피처 단위로 작업하되 중간 확인을 하고 싶을 때는 Cursor Agent Mode 같은 중간 지대가 맞습니다. 태스크나 레포 전체를 맡기고 싶을 때, 배치성 작업이나 자동화가 필요할 때는 Claude Code 같은 순수 에이전트를 선택하세요. 사람의 역할이 코드를 직접 조작하는 것에서, 목표와 방향을 지시하는 감독자로 이동한다는 점, 이게 핵심입니다.

=== s15 | 비용 현실 — 얼마나 드는가 (877자) ===

=== s16 | 멀티모달 (796자) ===
여러분, 혹시 AI한테 글만 보내고 계신가요? 사실 Claude는 텍스트만 받는 게 아닙니다. 이미지, 스크린샷, PDF를 그대로 붙여넣으면, 읽고 분석하고 심지어 구현까지 해줍니다. 이걸 멀티모달이라고 하는데요, 특히 디자이너나 기획자분들께 정말 강력한 워크플로우가 생깁니다.

=== s17 | 온톨로지 정의 (891자) ===
자, 이제 이 강의에서 가장 중요한 전환점에 왔습니다. 온톨로지, 이 개념을 제대로 이해하면 여러분이 AI를 쓰는 방식이 완전히 달라집니다.

온톨로지라는 단어가 좀 낯설죠? 쉽게 표현하면 지식의 지도입니다. 단순한 메모나 문서가 아니라, "무엇이 무엇과 어떻게 연결되어 있는가"를 기계가 읽을 수 있는 형태로 구조화한 것입니다. 개념과 관계, 그리고 규칙을 체계적으로 정의한 것이라고 보면 됩니다.

이게 왜 중요한가를 설명하기 위해 비유를 하나 들어볼게요. 온톨로지가 없는 AI와 있는 AI의 차이는, 매번 처음부터 가르쳐야 하는 직원과 맥락을 이미 내재화한 전문 어시스턴트의 차이와 같습니다.

온톨로지가 없으면 어떻게 되냐? 매번 AI에게 설명을 해야 합니다. "우리 회사가 뭐 하는 곳이에요?", "이 고객이 누구예요?", "우리 팀에서 쓰는 용어가 어떻게 되죠?" 세션이 끝나면 모든 게 사라져버립니다. 다음 날 다시 처음부터 설명해야 하죠. 이게 지금 많은 분들이 AI를 쓰는 방식입니다.

반면에 온톨로지를 갖춘 AI는 완전히 다릅니다. 회사 구조, 제품, 고객, 용어, 관계, 규칙이 이미 구조화되어 있습니다. 그러면 여러분은 그냥 지시만 하면 됩니다. 설명이 필요 없어요. "캠페인 배너 만들어줘"라고 하면, 브랜드 컬러가 뭔지, 어떤 톤인지, 어떤 규칙이 있는지를 AI가 이미 알고 있는 겁니다.

그리고 한 가지 오해를 짚고 넘어가겠습니다. 온톨로지를 구현하는 형식은 자유입니다. CLAUDE.md가 될 수도 있고, Notion 데이터베이스가 될 수도 있고, 마크다운 파일이 될 수도 있습니다. 중요한 건 형식이 아니라, 구조화가 되어 있다는 사실입니다. 맥락은 매번 넣어줘야 하지만, 온톨로지는 한 번 구조화하면 에이전트가 스스로 참조하는 영구적인 지식 기반이 됩니다.

=== s18 | 온톨로지 4요소 (938자) ===
그렇다면 온톨로지는 구체적으로 어떻게 구성되어 있을까요? 네 가지 요소로 이루어져 있습니다. 개체, 관계, 속성, 규칙입니다. 브랜드 디자인 실무 예시를 들면서 설명해볼게요.

첫 번째, 개체입니다. 영어로는 Entity라고 합니다. 한마디로 "우리 세계에 존재하는 것들"의 목록입니다. 사람, 제품, 팀, 고객, 프로세스 같은 것들이죠. 브랜드 실무라면 "Sparta Club이라는 브랜드가 있고, 디자인팀이 있고, B2B 기업 고객이 있다" 이런 식으로 나열하는 겁니다.

두 번째, 관계입니다. Relation이라고 하는데요, 이게 온톨로지의 진짜 핵심입니다. 개체들이 어떻게 연결되어 있는지를 정의하는 거예요. "디자인팀은 브랜드를 관리한다", "캠페인은 브랜드에 종속된다", "고객은 팀에 배정된다" 같은 연결 구조입니다. 이 관계가 정의되면 AI가 개체들 사이의 맥락을 이해하게 됩니다.

세 번째, 속성입니다. Attribute인데요, 각 개체가 가진 구체적인 특성과 값입니다. 브랜드라면 "주 컬러는 #FF6B35이고, 톤은 에너지와 도전이며, 파스텔 계열은 금지"라는 식으로 정의합니다. 이 속성이 있어야 AI가 "그럼 이 배너에서 이 색깔은 안 되겠구나"를 스스로 판단할 수 있습니다.

네 번째, 규칙입니다. Rule인데요, "이럴 땐 이렇게 해야 한다"는 판단 기준과 제약, 워크플로우입니다. "SNS에 발행하기 전에는 승인이 필요하다", "로고를 변형해서는 안 된다", "캠페인을 시작하기 전에는 반드시 브리프가 선행되어야 한다" 같은 것들이죠. AI가 자율적으로 판단해야 할 때 따르는 기준이 됩니다.

이 네 가지가 왜 중요하냐면, 이것들이 모이면 AI가 "배너 만들어줘"라는 한 마디에 브랜드 컬러, 톤, 금지 규칙을 모두 자동으로 반영하는 수준이 되기 때문입니다. 반면에 이게 없으면 매번 다시 설명해야 합니다. 구조화가 AI 품질을 결정합니다.

=== s19 | RAG (999자) ===
이제 RAG라는 개념을 살펴볼게요. Retrieval-Augmented Generation, 검색과 생성의 결합이라고 번역할 수 있는데요, 이게 왜 필요한지부터 설명드리겠습니다.

LLM, 즉 대형 언어 모델의 가장 큰 약점이 뭔지 아세요? 학습 데이터 이후의 정보는 모른다는 겁니다. Claude나 GPT는 특정 시점까지의 데이터로 학습되어 있습니다. 그 이후 바뀐 것은 모르고, 애초에 내 회사의 내부 문서나 개인 메모 같은 것들은 학습한 적 자체가 없습니다. 그래서 "우리 환불 정책이 어떻게 돼?"라고 물으면 AI는 그냥 일반적인 답변을 할 수밖에 없는 거죠. 이게 할루시네이션, 즉 AI가 틀린 내용을 자신 있게 답하는 문제의 주요 원인 중 하나입니다.

RAG는 이 문제를 검색과 생성을 결합해서 해결합니다. 작동 방식을 설명해 드릴게요. 사용자가 질문을 하면, 먼저 그 질문을 벡터로 변환해서 벡터 데이터베이스에서 관련 문서를 검색합니다. 벡터 검색이라는 게 단순한 키워드 검색과 달리, 의미적으로 유사한 내용을 찾아냅니다. 도서관에서 "환불 정책 관련 문서"를 찾는데, 단어가 정확히 일치하지 않아도 의미가 비슷한 것들을 함께 가져오는 것처럼요. 그 검색 결과와 원래 질문을 합쳐서 LLM에게 전달하면, LLM은 그 문서를 바탕으로 정확한 답변을 생성합니다. 출처도 함께 제시할 수 있죠.

그래서 RAG가 있는 AI는 내 문서를 벡터 데이터베이스에 넣으면 실시간으로 검색해서 답변합니다. 그리고 온톨로지와 RAG를 결합하면 가장 강력해집니다. 온톨로지로 지식을 구조화하고, 그것을 벡터 데이터베이스에 저장한 뒤, RAG로 검색해서 활용하는 방식이죠. 내 지식이 AI의 장기 기억이 되는 겁니다.

비개발자도 사용할 수 있는 RAG 도구들이 이미 많습니다. Notion AI는 워크스페이스 내 문서를 검색하고, Cursor는 전체 코드베이스를 RAG로 활용합니다. 그리고 지금 다음 슬라이드에서 소개할 Obsidian은 로컬에서 가장 강력한 RAG 지식 저장소가 됩니다.

=== s20 | Obsidian (988자) ===
RAG를 위한 지식 저장소로 Obsidian을 왜 추천하는지 설명드리겠습니다. 세 가지 이유가 있는데요, 각각이 정말 중요합니다.

첫 번째 이유, 순수 마크다운입니다. Obsidian의 모든 노트는 점md 파일, 즉 마크다운 파일로 저장됩니다. 이게 왜 중요하냐면, AI가 가공 없이 바로 읽을 수 있기 때문입니다. LLM이 가장 잘 이해하는 포맷이 바로 마크다운이거든요. Notion은 어떻냐고요? Notion은 독점 포맷이라서 API 없이 AI가 직접 접근할 수 없습니다. 반면 Obsidian은 그냥 폴더에 있는 텍스트 파일이에요. Claude Code나 다른 AI 에이전트가 경로만 알면 바로 읽어갑니다.

두 번째 이유, 양방향 링크가 관계를 정의합니다. Obsidian에서는 대괄호 두 개로 노트끼리 연결할 수 있습니다. 예를 들어 "고객사A" 노트에서 "프로젝트X"로 링크를 걸면, AI가 "고객사A는 프로젝트X와 연결되어 있다"는 관계를 자동으로 파악합니다. 아까 온톨로지의 4요소에서 배운 관계, Relation이 링크 하나로 자연스럽게 정의되는 거예요. 별도로 스키마를 설계할 필요 없이, 쓰면서 자연스럽게 온톨로지가 쌓이는 구조입니다.

세 번째 이유, 그래프 뷰가 온톨로지를 시각화해줍니다. Obsidian의 그래프 뷰를 켜면, 노트들이 서로 어떻게 연결되어 있는지를 한눈에 볼 수 있습니다. 이게 바로 온톨로지 섹션에서 설명한 지식 그래프가 눈앞에 펼쳐지는 겁니다. 어떤 지식이 고립되어 있는지, 어디가 허브인지, 지식의 빈 곳이 어디인지를 시각적으로 발견할 수 있습니다.

Notion과 비교해볼게요. Notion은 팀 협업과 프로젝트 관리에 강합니다. 함께 쓰는 데이터베이스, 칸반, 실시간 공동 편집이 탁월하죠. 그런데 개인 지식 축적과 AI RAG 연동 관점에서는 Obsidian이 앞섭니다. 그래서 전략적으로 둘 다 쓰는 것도 좋은 방법입니다. Notion으로 팀 작업을 하고, Obsidian에 나만의 온톨로지를 쌓는 방식으로요.

=== s21 | 온톨로지 실전 (976자) ===
그럼 실제로 어떻게 시작하면 될까요? 네 단계로 나눠서 설명드리겠습니다.

1단계는 수집입니다. 일단 뭐든 Obsidian Vault에 던져넣는 거예요. 회의 메모, 리서치 노트, 아이디어, PDF 클리핑, 뭐든 좋습니다. 완벽한 구조를 먼저 설계하려고 하면 절대 시작을 못 합니다. 일단 넣는 것부터 시작하세요.

2단계가 핵심입니다. 연결입니다. 대괄호로 관련 노트끼리 링크를 걸고, 태그를 달고, MOC, 즉 Map of Content라는 허브 노트를 작성합니다. 연결이 온톨로지를 만드는 겁니다. 수집은 원재료고, 연결이 진짜 지식을 만드는 과정이에요.

3단계는 구조화입니다. YAML 프론트매터로 메타데이터를 붙이고, 폴더 계층을 정리하고, Dataview 플러그인으로 쿼리를 걸면 AI가 검색하기 훨씬 좋은 형태가 됩니다.

4단계는 AI 연동입니다. 여기서 세 가지 방법이 있는데요, 상황에 맞게 고르시면 됩니다. 가장 강력한 방법은 MCP 서버 연결입니다. Obsidian Vault를 MCP 서버로 노출하면 Claude Code 같은 에이전트가 노트를 직접 읽고, 검색하고, 수정할 수 있습니다. 두 번째는 로컬 파일 직접 참조인데요, 사실 Obsidian Vault는 그냥 폴더입니다. CLAUDE.md에 Vault 경로를 한 줄 적어두면 AI가 그냥 읽어갑니다. 설정이 없어도 되는 가장 빠른 시작법이에요. 세 번째는 Obsidian 안에 있는 Smart Connections 플러그인입니다. 이걸 설치하면 노트를 자동으로 벡터 임베딩하고, Obsidian 안에서 AI 채팅으로 Vault 내용을 질문할 수 있습니다. 외부 도구 없이 Obsidian 자체가 RAG 시스템이 되는 거죠.

한 가지 강조하고 싶은 건, 구조보다 시작이 먼저라는 겁니다. 완벽한 폴더 구조를 설계하다 보면 결국 아무것도 못 씁니다. 일단 메모를 넣고, 나중에 링크와 태그와 MOC를 점진적으로 붙여나가세요. 구조는 쓰면서 자연스럽게 드러납니다.

=== s22 | RLHF → RLVR (1029자) ===
이제 AI 학습 패러다임이 어떻게 바뀌고 있는지, 그리고 그게 우리에게 왜 중요한지를 이야기해보겠습니다.

이 슬라이드의 핵심 메시지를 먼저 드릴게요. "AI를 얼마나 잘 쓰느냐"가 아니라, "AI가 얼마나 나를 잘 아느냐"가 격차를 만든다는 겁니다. 이 말이 무슨 뜻인지, RLHF와 RLVR의 차이를 통해 설명드리겠습니다.

먼저 RLHF, 인간 피드백 강화학습입니다. ChatGPT와 Claude가 초기에 사용한 핵심 학습 방식인데요, 작동 방식은 이렇습니다. AI가 출력물을 만들면, 사람이 A와 B 중에 어느 게 더 좋은지 선택합니다. 그 선호를 학습하는 거예요. 이 방식의 문제는 사람의 주관에 의존한다는 겁니다. "좋은 답변"이 뭔지 평가하는 기준이 사람마다 다르고, 속도도 느리고, 비용도 비쌉니다.

그래서 등장한 게 RLVR입니다. 검증 가능한 결과 강화학습, Reinforcement Learning from Verifiable Rewards입니다. 이름에 힌트가 있죠? 검증 가능한 결과로 학습한다는 겁니다. 코드가 실제로 실행되는가, 수학 답이 맞는가, 테스트를 통과하는가, 이런 객관적인 기준으로 자동 평가합니다. 사람이 개입하지 않아도 됩니다. AI가 스스로 초고속으로 반복하면서 개선할 수 있어요. DeepSeek, o1, Gemini 2.5의 추론 능력이 갑자기 급격히 좋아진 배경이 바로 이겁니다.

그런데 이게 우리한테 왜 중요하냐고요? RLVR로 인해 모델의 범용 능력이 빠르게 상향 평준화되고 있습니다. 누구나 같은 Claude, 같은 GPT를 씁니다. 모델 자체로는 차별화하기가 점점 어려워진다는 얘기예요.

그렇다면 진짜 차별화는 어디서 오느냐? 같은 모델에 어떤 컨텍스트와 온톨로지를 결합하느냐입니다. 내가 6개월 동안 쌓아온 온톨로지, 내 업무 방식과 도메인 지식을 담은 구조는 다른 사람이 복제할 수 없습니다. 이게 복리로 벌어지는 경쟁 우위입니다. 잘 구축된 온톨로지는 조직의 암묵지를 명시지로 전환하는 작업이기도 합니다. AI가 없어도 가치 있고, AI와 결합하면 레버리지가 극대화됩니다.

=== s23 | 컨텍스트 축적 (1024자) ===
마지막 슬라이드입니다. 왜 지금 당장 온톨로지를 시작해야 하는지, 그 당위성을 정리해드리겠습니다.

핵심 문장은 이겁니다. 모델은 공유된다. 온톨로지는 나만의 것이다. 같은 AI를 써도 결과가 극적으로 다른 이유가 바로 여기에 있습니다.

네 가지 포인트를 하나씩 짚겠습니다. 첫 번째, 일찍 쌓을수록 유리합니다. 지금 구조화한 지식이 6개월 후 AI 에이전트의 품질을 결정합니다. 복리 효과를 생각해보세요. 지금 시작하면 6개월 뒤에 남들보다 훨씬 정교한 AI를 갖게 됩니다. 반대로 6개월 뒤에 시작하면, 그만큼 격차를 메워야 합니다.

두 번째, 피드백 루프가 핵심입니다. AI가 틀리거나 맥락을 놓칠 때마다 온톨로지를 갱신하면, 같은 실수를 반복하지 않습니다. RLVR이 모델을 자동으로 개선하듯이, 온톨로지 갱신이 여러분의 AI를 개선합니다. 쓰면서 더 똑똑해지는 AI가 만들어지는 거죠.

=== s28 | 시작 가이드 (757자) ===
자, 이제 실전입니다. 직접 시작하는 방법을 알려드리겠습니다.

제가 추천하는 조합은 Claude Code와 oh-my-claudecode입니다. Claude Code는 Anthropic이 공식으로 만든 CLI 에이전트입니다. oh-my-claudecode는 그 위에 올라가는 멀티 에이전트 오케스트레이션 레이어입니다. GitHub 스타 1만 개를 넘은 오픈소스 프로젝트입니다. 32개의 특화 에이전트와 스마트 모델 라우팅을 제공합니다. 제가 직접 쓰는 조합이고, 이 강의를 만드는 데도 사용했습니다.

oh-my-claudecode의 핵심은 5가지 실행 모드입니다. autopilot은 아이디어를 받아 코드까지 완전히 자율 실행합니다. ralph는 완료될 때까지 verify와 fix를 반복합니다. ulw는 ultrawork의 줄임말로 최대 병렬 실행입니다. slash team은 여러 에이전트를 공유 태스크 리스트로 조율합니다. ccg는 Claude, Codex, Gemini 세 모델을 동시에 오케스트레이션하는 고급 모드입니다.

시작하기 전에 세 가지를 준비하셔야 합니다.

첫 번째는 Anthropic 구독입니다. Claude Code Pro는 월 20달러, Max는 월 100달러입니다. Opus 모델을 자주 쓰신다면 Max를 권장합니다.

두 번째는 Node.js입니다. Claude Code CLI 설치에 필요합니다.

세 번째는 Git입니다. 코드 버전 관리를 위해 필수입니다. macOS를 쓰신다면 이미 설치되어 있는 경우가 많습니다.

=== s29 | 설치 (710자) ===

두 번째 단계는 oh-my-claudecode 설치입니다. Claude Code를 실행한 상태에서 슬래시 plugin marketplace add oh-my-claudecode를 입력합니다. 또는 더 간단하게 슬래시 setup을 실행하면 설치 마법사가 안내해 줍니다.

세 번째 단계는 Anthropic 인증입니다. claude auth login을 입력합니다. Pro 또는 Max 구독 계정으로 OAuth 로그인을 진행하시면 됩니다.

네 번째 단계가 가장 중요합니다. CLAUDE.md 초기화입니다. 프로젝트 폴더에서 claude를 실행하고, CLAUDE.md 파일을 작성합니다. 이 파일은 AI에게 프로젝트의 맥락, 규칙, 금지사항을 선언하는 파일입니다. 이 파일을 잘 작성할수록 AI의 결과물 품질이 올라갑니다. 많은 분들이 이 단계를 소홀히 하는데, 실제로는 가장 핵심적인 단계입니다.

여기까지 따라오셨으면 잘 준비하신 겁니다. 막히는 부분이 있으면 각 도구의 공식 문서를 참조하시거나 커뮤니티에 질문하시면 빠르게 해결됩니다.

=== s30 | 실행 모드와 바이브 코딩 (970자) ===
설치가 완료됐으면, 이제 실제로 어떻게 쓰는지 알아보겠습니다.

화면 위쪽을 보시면 5가지 실행 모드가 있습니다.

autopilot은 가장 많이 쓰게 될 모드입니다. 아이디어를 주면 코드까지 완전히 자율 실행합니다.

ralph는 완료될 때까지 멈추지 않는 모드입니다. 검증하고, 실패하면 수정하고, 다시 검증하는 루프를 자동으로 반복합니다.

ulw는 ultrawork의 줄임말입니다. 최대 병렬 실행으로 여러 작업을 동시에 처리합니다.

슬래시 team은 N명의 에이전트를 공유 태스크 리스트에 붙여 협업시킵니다.

ccg는 Claude, Codex, Gemini 세 모델을 동시에 오케스트레이션하는 고급 모드입니다.

화면 아래쪽은 바이브 코딩 프로세스 6단계입니다. 이게 핵심입니다.

1단계 준비에서는 CLAUDE.md를 작성합니다. 프로젝트 맥락과 규칙을 선언하고, 공식 문서를 컨텍스트에 등록해 환각 현상을 방지합니다.

2단계 플래닝이 가장 중요합니다. ultrathink로 깊은 추론을 활성화하고, 요구사항을 구체화합니다. 단계를 분해하고 엣지케이스를 정의합니다. 전체 시간의 90%를 여기에 쓰세요.

3단계는 플랜 점검입니다. "플랜 출력해줘"라고 말해 AI가 세운 계획을 먼저 확인합니다. 방향이 맞는지 검증한 뒤에 구현을 시작합니다. 이 멈춤이 나중에 큰 시간을 아껴줍니다.

4단계는 구현입니다. autopilot 또는 ralph로 자율 실행합니다. 서브에이전트가 리서치, 코드, 테스트를 분리해서 처리합니다. 슬래시 team으로 병렬 조율도 가능합니다.

5단계는 개선입니다. ralph의 verify fix 루프로 자동 검증합니다. 문서를 업데이트하고 다음 페이즈로 반복하며 품질을 높입니다.

6단계는 커밋입니다. 변경사항을 리뷰하고 컨벤셔널 커밋 형식으로 커밋합니다. 다음 세션을 위해 CLAUDE.md도 갱신합니다.

이 6단계 루프를 조여 나가면, 혼자서도 팀 수준의 결과물을 만들 수 있습니다.
