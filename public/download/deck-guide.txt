 Part 1. LLM이란 무엇인가
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LLM은 Large Language Model의 약자로, 한국어로는 "대규모 언어 모델"이라고
부릅니다. ChatGPT, Claude, Gemini 등이 모두 LLM입니다.

  핵심 원리는 단 하나:
  "다음에 올 가장 자연스러운 단어를 확률적으로 예측하는 것"

이것이 번역, 요약, 코드 작성, 추론 등 모든 능력의 근간입니다. AI가 지식을
저장한 것이 아니라 "패턴"을 학습한 것이라는 점이 중요합니다.


[토큰화 — 텍스트를 숫자로 바꾸기]

AI는 글자를 그대로 읽지 못합니다. 먼저 텍스트를 "토큰"이라는 작은 조각으로
쪼개고, 각 조각에 고유한 숫자를 부여합니다. 예를 들어 "Claude is powerful"은
다섯 개의 토큰 [Cl][aude][is][power][ful]로 분해되고, 각각 숫자 ID
(3274, 8102, 318, 6244, 913)로 변환됩니다. 한국어는 영어보다 더 잘게
쪼개지기 때문에, 같은 내용이어도 토큰을 더 많이 소모합니다.


[벡터 임베딩 — 의미의 좌표]

토큰에 부여된 숫자는 단순한 일련번호가 아닙니다. 768개(혹은 그 이상)의
숫자로 이루어진 배열인데, 이것이 의미의 좌표 역할을 합니다. "강아지"와
"고양이"는 벡터 공간에서 가까이 위치하고, "자동차"는 멀리 있습니다.
심지어 "왕 - 남자 + 여자 = 여왕" 같은 의미 연산도 가능합니다.


[컨텍스트 윈도우 — AI의 책장]

컨텍스트 윈도우는 AI가 한 번에 참고할 수 있는 텍스트의 양입니다. 비유하자면
"AI가 한 번에 펼쳐볼 수 있는 책장"입니다. Claude의 경우 최대 200K 토큰(A4
약 500장 분량)까지 한 번에 처리할 수 있습니다. 이 책장 안에 든 내용만 참고할
수 있고, 밖으로 밀려난 정보는 존재하지 않는 것과 같습니다.


[어텐션과 스트리밍 출력]

AI는 "어텐션(Attention)" 메커니즘으로 각 단어가 다른 단어와 얼마나 관련
있는지를 계산합니다. 예를 들어 "그가 공을 찼다"에서 "그"와 "찼다"는 강한
연관을, "공을"과 "찼다"는 중간 연관을 가집니다. 이런 관계망을 수백 개
레이어에서 반복 계산해 문맥을 파악한 뒤, 가장 확률이 높은 다음 토큰을
하나씩 생성합니다. ChatGPT나 Claude에서 답변이 글자가 흘러나오듯 표시되는
이유가 바로 이 구조 때문입니다.


[할루시네이션 — AI의 본질적 한계]

AI는 확률 예측 방식이기 때문에, 모르는 것도 그럴듯하게 지어낼 수 있습니다.
이것을 "할루시네이션(Hallucination)"이라 부릅니다. 중요한 정보는 반드시
사람이 직접 검증해야 합니다.

[LLM 단독 vs AI 에이전트]

LLM 단독은 "질문하면 답변하고 끝"입니다. 기억도, 행동도 없습니다. 결과를
받아서 직접 실행하는 것은 사람의 몫입니다.

반면 AI 에이전트는 LLM에 도구들(파일 시스템, 터미널, 웹, API 등)을 연결한
것입니다. 에이전트는 스스로 계획을 세우고, 실행하고, 결과를 확인하고,
잘못되면 재시도합니다. 사람은 목표만 주면 됩니다.

에이전트를 구성하는 요소:
  - LLM: 판단과 추론을 담당하는 두뇌
  - 툴 호출: 파일 읽기/쓰기, 터미널 명령 실행, 웹 검색
  - 루프: 실행 → 결과 확인 → 필요시 재시도
  - 메모리: 대화의 맥락을 유지
  - MCP: 외부 앱(Notion, Slack 등)과 직접 연결하는 프로토콜


[GitHub 기초 — 왜 필요한가]

AI 에이전트로 코드를 작성하더라도, 그 코드를 안전하게 관리하려면 GitHub이
필요합니다. 핵심 개념 네 가지만 알면 충분합니다:

  - Repository(레포지토리): 프로젝트 전체가 담긴 상자
  - Commit(커밋): "여기까지 저장"하는 저장점. 언제든 되돌릴 수 있음
  - Branch(브랜치): 원본을 건드리지 않고 실험하는 공간
  - Pull Request(PR): 실험이 끝나면 원본에 합쳐달라는 요청


[개발의 기본 구조]

모든 앱은 세 층으로 이루어져 있습니다:
  - 프론트엔드: 사용자가 보는 화면 (HTML, CSS, JavaScript)
  - 백엔드: 눈에 안 보이는 서버 (API, 인증, 비즈니스 로직)
  - 데이터베이스: 데이터를 저장하고 꺼내는 곳


[API, CLI, MCP — 자주 나오는 세 가지 용어]

  - API (Application Programming Interface): 두 프로그램이 대화하는 규칙.
    비유하면 "식당의 메뉴판"입니다.
  - CLI (Command Line Interface): 텍스트 명령어로 컴퓨터를 조작하는 방식.
    비유하면 "직접 명령하는 리모컨"입니다.
  - MCP (Model Context Protocol): AI 에이전트가 외부 앱을 직접 읽고 쓸 수
    있게 해주는 프로토콜. 비유하면 "AI의 만능 어댑터"입니다.

[생태계 전체 관계도 — 위에서 아래로]

  AI 모델 (Claude, GPT-4o, Gemini, Llama)
       ↓
  코딩 에이전트 (Claude Code, Devin, Aider) — AI 주도, 자율 실행
       ↓
  에디터 내 에이전트 (Cursor Agent, Windsurf, Cline) — 에디터 안에 에이전트 탑재
       ↓
  에디터/IDE (Cursor, VS Code, Zed, JetBrains) — 사람 주도, AI가 보조
       ↓
  터미널 환경 (Tmux, iTerm2, Warp, zsh) — 에이전트 실행 기반


[진화 경로 — 6단계]

대부분의 사람이 이 순서로 AI 도구 활용이 깊어집니다:

  STAGE 1: 단독 LLM 문답 — ChatGPT에 질문하고 복붙
  STAGE 2: 노코드 자동화 — Zapier/Make로 앱 간 연결
  STAGE 3: Claude Code 입문 — AI가 파일을 직접 수정
  STAGE 4: AI-native 터미널 — 여러 세션 동시 운용
  STAGE 5: 멀티 에이전트 + 하네스 — 에이전트를 조율하는 에이전트
  STAGE 6: MCP — 외부 앱과 에이전트 직접 연결

 Part 4. Claude Code 핵심 개념과 바이브 코딩
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Claude Code는 단순한 명령어 도구가 아니라, 작업 흐름 전체를 설계하는
프레임워크입니다. 핵심 개념 6가지:

  - CLAUDE.md: 프로젝트 루트에 두는 AI 행동 지침서. 세션 시작 시 자동 로드.
  - 서브 에이전트: 메인 에이전트가 하위 작업을 별도 에이전트에 위임.
  - 커맨드(슬래시): /로 시작하는 단축 명령어.
  - 훅(Hooks): 특정 이벤트 발생 시 자동 실행되는 스크립트.
  - 스킬(Skills): 반복 작업을 파일로 정의한 재사용 가능한 지식 모듈.
  - MCP 서버: 외부 앱 연결 프로토콜.


[.claude/ 폴더 구조]

  CLAUDE.md — AI의 행동 매뉴얼 (200줄 이내 권장)
  rules/ — 관심사별 분리된 모듈 규칙
  skills/ — 자동 트리거 워크플로우
  commands/ — 수동 슬래시 커맨드
  agents/ — 전문 서브에이전트
  settings.json — 권한 및 훅 설정


[바이브 코딩 루프 — 5단계]

  PHASE 1 준비: CLAUDE.md 작성, 공식 문서 등록, 스킬 파일 세팅
  PHASE 2 플래닝 ★: 요구사항 구체화, 단계 분해, 엣지케이스 정의
  PHASE 3 플랜 점검: 출력 요청, 방향 확인, 수정 후 재확인
  PHASE 4 구현: AI가 코드 작성, 서브 에이전트 위임
  PHASE 5 점검·개선: 의도대로 됐는지 확인, 문서 업데이트 → PHASE 2 반복

  핵심 메시지: "플래닝이 90%다. 구현은 AI가 한다."


[안티패턴 — 흔한 실수 7가지]

  1. 플래닝 없이 바로 구현 요청 — 모호한 지시는 엉뚱한 결과로
  2. 컨텍스트 오염 — 같은 세션에서 너무 많은 수정 반복
  3. 무한 수정 루프 — "조금만 더"가 코드를 점점 복잡하게
  4. 검증 없이 배포 — AI 출력 = 초안이지 완성본이 아님
  5. 이해 못 한 채로 사용 — 블랙박스로 두면 버그 수정 불가
  6. 문서 업데이트 생략 — 다음 세션에서 맥락 단절
  7. 너무 큰 태스크를 한 번에 — 기능 단위로 잘게 쪼개서 진행

 Part 5. 하네스 — AI를 길들이는 구조
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

하네스(Harness)는 원래 "말에 채우는 고삐"를 뜻합니다. AI 에이전트에게도
같은 개념을 적용합니다: LLM의 범용 능력은 유지하면서, 내 프로젝트와 규칙에
맞게 방향을 잡아주는 장치가 하네스입니다.

  하네스 없으면: 매번 반복 설명, 스타일 불일치, 규칙 모름, 맥락 소멸
  하네스 있으면: CLAUDE.md로 맥락 전달, 스킬로 패턴 유지, MCP로 외부 연동


[코딩 하네스 실전 — 두 가지 도구]

  oh-my-claudecode (OMC): "건축 현장의 현장 감독"
    - 큰 작업을 32개 전문 에이전트에 자동 위임
    - autopilot(완전자율), ralph(완료까지 반복), ultrawork(병렬처리)
    - 간단한 일은 싼 모델, 어려운 일은 비싼 모델로 자동 분배

  oh-my-hwclaude: "레이저 가이드가 달린 정밀 공구"
    - 파일의 정확한 줄 번호+해시로 위치를 특정 (편집 성공률 6.7%→68.3%)
    - 편집 실패 시 자동 복구
    - 들여쓰기 깨짐 같은 흔한 실수 자동 보정


[비즈니스 하네스 — AI에게 부서를 만들어주기]

코딩뿐 아니라 비즈니스에도 하네스를 적용할 수 있습니다:
  - 에이전트 = 전문 부서장 (마케팅, 영업, 법무, 재무 등 16명)
  - 플러그인 = 업무 매뉴얼 (17개 세트)
  - 스킬 = 매뉴얼 안의 구체적 절차서 (110개)
  - 라우팅 = 안내 데스크 (요청을 적절한 부서에 전달)
  - 커맨드 = 단축 버튼 (/ask, /route 등)
  - 훅 = 자동 트리거 (세션 종료 시 작업 기록 저장 등)


[하네스 성숙도 — 4단계]

  LV.0 무장비: ChatGPT에 질문만. 매번 처음부터 설명.
  LV.1 기본 제어: CLAUDE.md 작성. 프로젝트 맥락 전달.
  LV.2 도구 연결: MCP로 외부 연동, 스킬·규칙 추가.
  LV.3 자율 순환: 온톨로지 + RAG + 피드백 루프 운영.

  핵심: 모델은 공유된다. 하지만 어떤 하네스를 씌웠느냐에 따라 결과는
  완전히 다르다. 잘 만든 하네스는 복제 불가능한 경쟁 우위다.


[에디터 vs 에이전트 — 핵심 차이]

  에디터(VS Code): 줄/함수 단위 작업, 매 스텝 사람 개입, 사람 주도
  에디터 내 에이전트(Cursor): 피처 단위, 승인만, 사람→AI 전환
  순수 에이전트(Claude Code): 태스크/레포 단위, 시작·끝만, AI 주도

 Part 7. 온톨로지 — AI의 장기 기억

[온톨로지란?]

개념, 관계, 규칙을 구조적으로 정의한 "지식의 지도"입니다. 단순한 메모가
아니라, "무엇이 무엇과 어떻게 연결되어 있는가"를 기계가 읽을 수 있는 형태로
정의한 것입니다. 맥락은 매번 넣어줘야 하지만, 온톨로지는 한 번 구조화하면
에이전트가 스스로 참조하는 영구적인 지식 기반이 됩니다.


[온톨로지 4가지 구성 요소]

  1. Entity(개체): 존재하는 것들의 목록 — 사람, 제품, 팀, 고객
  2. Relation(관계): 개체들이 어떻게 연결되는지 — "A는 B에 속한다"
  3. Attribute(속성): 각 개체의 특성·값 — 브랜드 컬러, 톤앤매너
  4. Rule(규칙): 판단 기준·워크플로우 — "이럴 땐 이렇게 해야 한다"


[암묵지와 명시지]

  명시지(Explicit Knowledge): 이미 문서화된 지식
    매뉴얼, 가격표, 조직도 → AI가 바로 읽을 수 있음

  암묵지(Tacit Knowledge): 머릿속에만 있는 지식
    "이 클라이언트는 이렇게 대해야 해" 같은 경험, 직관, 감각
    → 이것을 꺼내서 구조화하는 것이 온톨로지의 진짜 가치

  중요한 경고: 규모가 커지면 하고 싶어도 못 합니다. 3명일 때는 가능하지만,
  30명이면 어렵고, 300명이면 불가능합니다. 핵심 인력이 퇴사하면 그 사람의
  암묵지는 영원히 사라집니다. 지금, 작을 때 구조화해야 합니다.


[AI Native — 암묵지를 안 만드는 전략]

가장 좋은 전략은 암묵지가 처음부터 생기지 않게 하는 것입니다. 모든 일을
AI를 통해 하면, 과정과 판단이 자동으로 기록됩니다. AI에게 지시하는 순간
암묵지가 명시지로 전환됩니다.

  과거: 모든 일을 손으로 했다
  현재: 모든 일을 컴퓨터로 한다
  다음: 모든 일을 AI로 한다

컴퓨터 없이 일하는 게 상상이 안 되듯, AI 없이 일하는 것도 곧 그렇게 됩니다.


[RAG — AI가 모르는 것을 검색해서 답하는 방법]

RAG(Retrieval-Augmented Generation)는 검색과 생성을 결합한 방법입니다.

  사용자 질문 → 벡터DB에서 관련 문서 검색 → 질문+검색 결과 조립
  → LLM이 답변 생성 → 문서 기반 정확한 답변(출처 포함)

온톨로지로 지식을 구조화 → 벡터DB에 저장 → RAG로 검색하면,
내 지식이 AI의 장기기억이 됩니다.


[Obsidian — AI 온톨로지에 최적화된 도구]

  이유 1: 순수 마크다운 — AI가 가공 없이 바로 읽는 포맷
  이유 2: 양방향 링크 = 관계 정의 — [[링크]] 하나로 온톨로지 관계 생성
  이유 3: 그래프 뷰 = 온톨로지 시각화 — 지식 구조가 눈앞에 펼쳐짐

AI 연동 방법:
  - METHOD 1 (추천): MCP 서버 연결 — 에이전트가 노트를 직접 읽고 검색
  - METHOD 2 (간단): 로컬 파일 직접 참조 — 경로만 지정
  - METHOD 3 (Obsidian 내장): Smart Connections 플러그인 — Obsidian 자체가 RAG


[보안과 프라이버시]

  - 데이터가 어디로 가는지: Claude API는 학습에 사용 안 함. 하지만 모든
    서비스가 그렇진 않으므로 이용약관 반드시 확인.
  - 민감 정보 구분: 고객 개인정보, 계약 금액, 내부 전략은 구분해서 관리.
  - 로컬 우선 전략: Obsidian은 내 컴퓨터에만 저장, Claude Code는 로컬 실행.
  - 조직 도입 시: 데이터 거버넌스 정책을 먼저 수립.

 Part 8. RLHF/RLVR과 Eval

[RLHF → RLVR: 쓸수록 똑똑해지는 AI]

  RLHF(인간 피드백 강화학습): 사람이 A/B를 비교해 선호를 학습.
    느리고 비싸며 사람의 주관에 의존.

  RLVR(검증 가능한 결과 강화학습): 코드 실행, 수학 정답, 테스트 통과
    같은 객관적 기준으로 자동 평가. 사람 없이도 초고속 자기개선 가능.

  핵심: "AI를 얼마나 잘 쓰느냐"가 아니라 "AI가 얼마나 나를 잘 아느냐"가
  격차를 만든다.


[Eval — AI 성능을 측정하는 과학]

  Level 1 — Model Eval: 모델 자체의 능력 측정 (벤치마크)
  Level 2 — Task Eval: 프롬프트에 따른 결과 비교 (A/B 테스트)
  Level 3 — System Eval: 에이전트 워크플로우 전체 평가

  측정할 수 없으면, 개선할 수 없습니다.


[Eval-Driven Development (EDD)]

코드에서 TDD(테스트 주도 개발)가 있듯, AI에도 EDD가 있습니다:
  1. 먼저 평가 기준을 만들고
  2. AI 출력을 측정하고
  3. 기준을 통과할 때까지 개선하는 방식

[컨텍스트 축적이 경쟁 우위인 이유]

  1. 모델은 공유된다: 누구나 같은 Claude, 같은 GPT를 쓴다.
     하지만 온톨로지는 나만의 것.
  2. 일찍 쌓을수록 유리하다: 지금 구조화한 지식이 6개월 후 AI 품질을 결정.
  3. 피드백 루프가 핵심이다: 틀릴 때마다 갱신하면 같은 실수를 반복하지 않음.
  4. 조직이 AI와 함께 학습한다: 온톨로지 갱신이 조직의 AI를 개선.


[지금 당장 시작하는 로드맵]

  MONTH 1: 자주 쓰는 개체·관계·규칙을 구조화
  ONGOING: AI가 틀릴 때마다 온톨로지 갱신


[진짜 경쟁 우위(해자)는 무엇인가]

    풀어야 하는지는 사람이 정한다. 이것이 가장 확실한 해자.
  MOAT #2 온톨로지 + 독점 데이터: 같은 AI라도 내 데이터가 있으면 완전히 다름.
  MOAT #3 지속 학습: 피드백 루프로 계속 개선되는 구조.


[전문가들의 시각]

  Sequoia/a16z: 네트워크 효과 + 독점 데이터가 해자
  Peter Thiel: 경쟁하지 않는 것 = 독점이 해자
  Marc Andreessen: 배포 능력(Go-to-Market)이 해자
  Sam Altman: AGI에 가까워질수록 남는 것은 신뢰와 관계

  종합: AI 능력 자체는 해자가 아니다. 해자는 AI 위에서 쌓이는 것들.

[글쓴이가 펜을 버렸다고, 그의 글이 사라지지 않는다]

종이 → 타자기 → 워드프로세서 → AI. 도구는 계속 바뀌었지만, 무엇을 쓸지
결정한 것은 언제나 사람이었습니다.

도구의 진화 타임라인:
  펜·종이 (~1900s): 인쇄가 특권이었다
  타자기·워드 (1990s): 컴활 자격증이 스펙이었다
  컴퓨터·인터넷 (2000s~): 지금은 공기처럼 당연
  AI 에이전트 (지금 → 곧): 곧 공기처럼 당연해진다

도구가 바뀌어도 바뀌지 않은 것:
  무엇을 말할지, 왜 말할지, 누구를 위한지.


[도구는 같다. 격차는 어디서 오는가]

둘 다 갖고 있는 것: 컴퓨터, AI 도구, 인터넷
격차를 만드는 것: 어떤 문제를 볼 것인가, 무엇을 알고 있는가, 무엇을 만들려 하는가

  과거: "컴퓨터를 쓴다" — 한때 스펙이었지만 지금은 공기.
  현재: "AI를 쓴다" — 빠르게 평준화된다. 기본기를 갖추는 것.
  언제나: "본질을 안다" — 무엇을 만들지, 왜 만드는지. AI가 대신할 수 없다.
