01 · LLM이란 무엇인가

LLM이란 무엇인가

Large Language Model — 대규모 언어 모델

인터넷의 방대한 텍스트를 학습해서 "다음에 올 가장 자연스러운 단어"를 확률적으로 예측하는 모델. 번역·요약·코드 작성·추론이 모두 이 하나의 원리에서 나온다. 지식을 저장한 게 아니라, 패턴을 학습한 것.

STEP 1 · 토큰화 — 텍스트를 숫자로

AI는 텍스트를 그대로 읽지 않는다. 먼저 토큰이라는 단위로 쪼개고, 각 토큰을 고유한 숫자 ID로 변환한다.

토큰은 단어 전체가 아닌 의미 단위 조각. 한국어는 음절 단위로 더 잘게 쪼개져 같은 내용이어도 토큰 수가 더 많이 소모된다.

"Claude is powerful" ↓ 토큰 분해 [Cl] [aude] [is] [power] [ful] ↓ 숫자 ID 변환 3,274 · 8,102 · 318 · 6,244 · 913 ↓ 벡터 임베딩 [0.82, -0.31, 0.57 ... x768차원]

학습

패턴 학습

수천억 개 텍스트에서 언어 패턴을 학습. 사실 저장이 아닌 확률 분포를 학습한다.

추론

토큰 예측

다음 토큰의 확률 분포를 계산하고 가장 자연스러운 답을 한 글자씩 생성한다.

제약

할루시네이션

모르는 것도 그럴듯하게 지어낼 수 있다. 중요한 정보는 반드시 검증이 필요하다.

STEP 1-2 · 벡터 임베딩 + 컨텍스트 윈도우

의미의 좌표와 AI의 책장

벡터 임베딩(Vector Embedding)

숫자 배열은 단순한 ID가 아니다. 각 숫자는 의미의 좌표다. "강아지"와 "고양이"는 벡터 공간에서 가까이 위치하고, "자동차"는 멀리 있다.

• 의미가 같으면 가깝다: "강아지"와 "개"는 거의 같은 위치

• 벡터 연산이 가능: 왕 - 남자 + 여자 ≈ 여왕

• 온톨로지 + 벡터DB(RAG): 지식을 벡터로 저장하면 AI가 의미 기반 검색 가능

STEP 2 · 컨텍스트 윈도우

AI의 책장

컨텍스트 윈도우는 AI가 한 번에 펼쳐볼 수 있는 책장이다. 책장 안에 든 것만 참고할 수 있고, 책장 밖은 존재하지 않는 것과 같다. 대화가 길어질수록 오래된 내용이 밖으로 밀려난다.

Claude 200K = A4 약 500장 분량

STEP 3 · 4 · 추론과 출력

어텐션 추론과 스트리밍 출력

STEP 3 · 어텐션 그래프로 의미 연결

각 토큰은 다른 모든 토큰과 "얼마나 관련 있는가"를 계산한다. 이 관계망을 수백 개 레이어에서 반복 계산해 문맥 속 의미를 추론한다.

STEP 4 · 토큰 하나씩 생성 → 스트리밍

추론 결과로 다음에 올 토큰의 확률 분포가 나온다. 가장 높은 확률의 토큰을 선택 → 컨텍스트에 추가 → 다시 추론. 이 과정을 반복하며 한 글자씩 생성.

ChatGPT·Claude에서 답변이 흘러나오듯 표시되는 이유가 이 구조 때문이다.

02 · 에이전트 전환점

LLM과 AI 에이전트의 차이

LLM 단독

질문 → 답변. 끝.

입력이 들어오면 출력을 냄. 기억 없음. 행동 없음. 매번 처음부터. 사람이 결과를 받아서 직접 실행해야 함.

AI 에이전트

태스크 → 계획 → 실행 → 검증. 반복.

LLM에 툴(파일·터미널·웹·API)을 연결하면 에이전트가 됨. 스스로 계획하고, 실행하고, 결과를 보고 재시도. 사람은 목표만 주면 됨.

에이전트 = LLM + 이것들

∘ LLM — 판단·추론 엔진 ☆ 툴 호출 — 파일·터미널·웹 실행 ↻ 루프 — 실행→결과 확인→재시도 ▪ 메모리 — 컨텍스트 유지 &hexmark; MCP — 외부 앱 연결 프로토콜

개발의 기본 구조

프론트엔드 · 백엔드 · 데이터베이스 — 모든 앱은 이 세 층으로 이루어져 있다

프론트엔드 (클라이언트)

사용자가 보는 화면. HTML(구조), CSS(스타일), JavaScript(동작), React/Next.js(프레임워크)

백엔드 (서버)

눈에 안 보이는 뒷단. API(대화 창구), 인증, 비즈니스 로직, Node.js/Python/Go

데이터베이스 (DB)

데이터를 저장하고 꺼내는 곳. SQL, NoSQL, Supabase/Firebase, ORM

바이브 코딩 안티패턴

흔한 실수 7가지 — 미리 알면 절반은 피할 수 있다

01. 플래닝 없이 바로 구현 요청

"로그인 기능 만들어줘"처럼 모호하게 던지면 AI가 엉뚱한 방향으로.

02. 컨텍스트 오염

같은 세션에서 너무 많은 수정 반복하면 AI가 혼란.

03. 무한 수정 루프

"조금만 더 고쳐줘" — 일정 수준의 변경은 처음부터 다시 짜는 게 빠르다.

04. 검증 없이 배포

AI가 "완성됐다"고 해도 직접 확인해야 한다.

05. 이해 못 한 채로 사용

블랙박스로 두면 버그가 났을 때 고칠 수 없다.

06. 문서 업데이트 생략

다음 세션에서 AI가 이전 상황을 모른 채 시작한다.

07. 너무 큰 태스크를 한 번에

"전체 앱 만들어줘"처럼 큰 요청은 방향이 틀어지면 수정 범위가 너무 넓어진다. 기능 단위로 잘게 쪼개서 진행.

하네스 — AI 에이전트를 길들이는 구조

에이전트는 강력하다. 하지만 방향 없이는 쓸모없다.

하네스(Harness)는 원래 말에 채우는 고삐·마구를 뜻한다. AI 에이전트도 마찬가지 — LLM의 범용 능력은 유지하면서 내 프로젝트·도메인·규칙에 맞게 제어하는 장치가 하네스다.

하네스 없는 에이전트 = 고삐 없는 말

  • 매번 "우리 프로젝트는..." 반복 설명
  • 코딩 스타일이 매번 달라짐
  • 금지 규칙을 모름
  • 외부 도구 접근 불가
  • 세션이 바뀌면 맥락 소멸

하네스 있는 에이전트 = 조련된 경주마

  • CLAUDE.md — 프로젝트 맥락 자동 전달
  • 스킬 파일 — 일관된 코딩 패턴
  • 규칙 — 금지 사항 강제
  • MCP — DB·API·외부 도구 직접 조작
  • 온톨로지 — 장기 기억

코딩 하네스 실전

건축 현장에 비유하면 — 현장 감독 + 정밀 도구 세트

oh-my-claudecode (OMC)

비유: 건축 현장의 현장 감독

• 큰 작업을 32개 전문 에이전트에 자동 위임
• autopilot(완전자율), ralph(완료까지 반복), ultrawork(병렬처리)
• 간단한 일은 싼 모델(Haiku), 어려운 일은 비싼 모델(Opus)

oh-my-hwclaude

비유: 레이저 가이드가 달린 정밀 공구

• 파일의 정확한 줄 번호+해시로 위치 특정 (성공률 6.7%→68.3%)
• 편집 실패 시 자동 복구
• 들여쓰기 깨짐 등 흔한 실수 자동 보정

비즈니스 하네스 실전

대기업에 전문 부서 시스템이 있듯, AI에게도 부서를 만들어줄 수 있다

에이전트 = 전문 부서장

마케팅·영업·법무·재무·HR 등 16명의 부서장. 각자 도메인 전문 지식.

플러그인 = 업무 매뉴얼

각 부서가 보유한 매뉴얼 묶음. 17개 세트.

스킬 = 매뉴얼 안의 절차서

'경쟁사 분석하는 법', 'PRD 작성하는 법' 같은 110개의 구체적 실행 절차.

하네스 성숙도

어디까지 왔는가 — 4단계로 보는 에이전트 제어 수준

LV.0

무장비

ChatGPT에 질문만. 매번 처음부터 설명.

LV.1

기본 제어

CLAUDE.md 작성. 프로젝트 맥락 전달.

LV.2

도구 연결

MCP로 외부 연동. 스킬 파일·규칙 추가.

LV.3

자율 순환

온톨로지 + RAG. 피드백 루프 운영.

하네스 = 에이전트의 진짜 차별화 포인트. 모델은 공유된다 — 누구나 같은 Claude, 같은 GPT를 쓴다. 하지만 어떤 하네스를 씌웠느냐에 따라 결과는 완전히 다르다. 잘 만든 하네스는 곧 복제 불가능한 경쟁 우위다.

온톨로지 — AI의 장기 기억을 설계한다

맥락을 매번 넣는 것과 온톨로지를 구축하는 것은 다르다

온톨로지(Ontology)란?

개념·관계·규칙을 구조적으로 정의한 지식의 지도. 단순한 메모나 문서가 아니라, "무엇이 무엇과 어떻게 연결되어 있는가"를 기계가 읽을 수 있는 형태로 정의한 것. 한 번 구조화하면 에이전트가 스스로 참조하는 영구적인 지식 기반이 된다.

온톨로지 없는 AI

매번 처음부터 가르쳐야 하는 직원. 세션이 끝나면 모두 사라짐.

온톨로지 갖춘 AI

맥락을 내재화한 전문 어시스턴트. 지시만 하면 됨. 설명은 필요 없음.

온톨로지를 구성하는 4가지 요소

Entity, Relation, Attribute, Rule

ENTITY · 개체

존재하는 것들의 목록

사람, 제품, 팀, 고객, 프로세스 등

RELATION · 관계

개체들이 어떻게 연결되는지

"A는 B에 속한다", "A는 B를 담당한다"

ATTRIBUTE · 속성

각 개체가 가진 특성·값

브랜드 컬러, 톤앤매너, 타겟, 제약사항

RULE · 규칙

판단 기준·제약·워크플로우

"이럴 땐 이렇게 해야 한다"는 로직

암묵지와 명시지

온톨로지의 진짜 어려움 — 머릿속에만 있는 지식을 꺼내는 것

명시지 (Explicit Knowledge)

매뉴얼, 가격표, 조직도, 계약서. 이미 문서화되어 있어 AI가 바로 읽을 수 있음. 온톨로지 구축 난이도: 낮음

암묵지 (Tacit Knowledge)

"이 클라이언트는 이렇게 대해야 해". 경험, 직관, 맥락 판단, 감각. 온톨로지 구축 난이도: 매우 높음

규모가 커지면 하고 싶어도 못 한다. 3명일 때 암묵지를 꺼내는 건 가능합니다. 30명이 되면 이미 어렵습니다. 300명이 되면 불가능합니다. 지금, 작을 때, 꺼낼 수 있을 때 구조화해야 합니다.

암묵지를 안 만드는 것

가장 좋은 온톨로지 전략은 암묵지가 생기지 않게 하는 것이다

모든 일을 AI를 통해 하면, 과정과 판단이 자동으로 기록됩니다.

머릿속에서 판단→AI에게 지시→자동으로 기록=암묵지 발생 차단

AI Native = 모든 일을 AI로 하는 것이 자연스러운 상태

컴퓨터 없이 일하는 게 상상이 안 되듯, AI 없이 일하는 것도 곧 그렇게 됩니다.

RAG — AI가 모르는 것을 검색해서 답하는 방법

Retrieval-Augmented Generation — 검색과 생성의 결합으로 LLM의 지식 한계를 극복

LLM의 가장 큰 약점은 학습 데이터 이후의 정보를 모른다는 것. RAG는 질문이 들어오면 먼저 관련 문서를 벡터DB에서 검색해서 컨텍스트에 넣고, 그 내용을 바탕으로 답변을 생성한다.

사용자 질문→ 벡터DB 검색→ 컨텍스트 조립→ LLM 생성→ 정확한 답변

Obsidian — AI의 지식 저장소

로컬 마크다운 + 양방향 링크 + 지식 그래프 — RAG에 최적화된 개인 온톨로지 저장소

REASON #1

순수 마크다운

모든 노트가 .md 파일. AI가 가공 없이 바로 읽을 수 있다.

REASON #3

그래프 뷰 = 온톨로지 시각화

노트 간 연결을 그래프로 시각화 → 지식의 빈 곳을 발견.

08 · RLHF → RLVR

"AI가 얼마나 나를 잘 아느냐"가 격차를 만든다

RLHF

인간 피드백 강화학습

사람이 A/B를 비교해 선호를 학습. 느리고 비쌈. 사람의 주관에 의존.

RLVR

검증 가능한 결과 강화학습

코드 실행, 수학 정답, 테스트 통과 — 객관적 기준으로 자동 평가. 사람 없이도 초고속 자기개선.

Eval — AI 성능을 측정하는 과학

측정할 수 없으면, 개선할 수 없다

Level 1

Model Eval

모델 자체의 능력 측정. 벤치마크 비교.

Level 2

Task Eval

같은 모델이라도 프롬프트에 따라 결과가 다르다. A/B 비교, LLM-as-Judge.

Level 3

System Eval

에이전트 워크플로우 전체 평가. RAG 정확도, E2E 성공률.
